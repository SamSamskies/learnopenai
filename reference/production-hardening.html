Production Hardening

Auth, rate limits, and two error channels for streaming AI routes.

Two error channels

When Transport Client handles
Before stream starts (auth, rate limit, bad body) HTTP status + JSON { error } if (!res.ok) — no SSE parser
After stream started (OpenAI failure mid-run) SSE snapshot phase: "error" Reducer branch on phase

Auth on the route

// Server — never expose OPENAI_API_KEY to the browser
const token = req.headers.authorization?.replace(/^Bearer /, "");
if (token !== process.env.TRIAGE_API_SECRET) {
  return Response.json({ error: "Unauthorized" }, { status: 401 });
}

// Client — session cookie or short-lived app token, not the OpenAI key
fetch("/api/triage", {
  headers: { Authorization: `Bearer ${appToken}` },
});

Rate limiting

// Return 429 JSON before opening the SSE stream
if (overLimit(clientIp)) {
  return Response.json({ error: "Rate limit exceeded" }, { status: 429 });
}

In-memory buckets are fine for demos and single-instance deploys. Use Redis or your edge provider for multi-instance production.

Sanitize errors

Log (server) Send (client)
Full OpenAI error + stack Generic “Triage request failed”
Request id, user id, model phase: "error" message only
