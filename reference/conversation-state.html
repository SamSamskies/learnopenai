Conversation State

Three ways to carry context across turns in the Responses API.

Approach You send each turn Best for
previous_response_id Only the new user message (+ stable instructions) Standard chat threads; simplest server-managed state
Manual item replay Prior output items + new message in input Trimming context, custom state, store: false
Conversations API Messages on a durable conversation object Long-lived threads across sessions/devices 
const first = await client.responses.create({
  model: "gpt-4.1-nano",
  input: "My favorite color is teal.",
  store: true,
});

const second = await client.responses.create({
  model: "gpt-4.1-nano",
  input: "What is my favorite color?",
  previous_response_id: first.id,
  store: true,
});
Gotcha Detail
store: true Required for server-side chaining (default). Set store: false only when you manage state yourself.
instructions Do not carry over via previous_response_id. Resend stable system prompts every turn.
Billing Prior context still counts as input tokens each turn (prompt caching discounts repeated prefixes).
Retention Stored responses kept ~30 days unless deleted. Conversations API persists longer.
Save the id Store response.id (resp_…) after each turn — from blocking calls or response.completed when streaming.

Chat product pattern

Per session → keep lastResponseId in memory or your DB
Each user message → input: newMessage, previous_response_id: lastResponseId
On complete → lastResponseId = response.id
New chat → clear id, start fresh with no previous_response_id
