Background Mode

Lesson 2 held the connection open while tokens streamed. Some jobs outlive your HTTP timeout — background mode lets them finish anyway.

A sync responses.create blocks until the model finishes (Lesson 1). Streaming keeps the socket alive with partial output (Lesson 2). But a long report, deep research pass, or heavy tool loop can exceed gateway timeouts (30–60s). Set background: true and poll instead. Background mode guide.

What changes

Detail
Returns immediately You get a response.id and a status — often queued or in_progress — not the final answer.
You poll responses.retrieve(id) until status leaves queued / in_progress.
Requires storage store: true — background requests without storage are rejected.
Not ZDR-safe Response data is retained ~10 minutes for polling. Skip background on zero-data-retention projects.

Start and poll

const started = await client.responses.create({
  model: "gpt-4.1-nano",
  instructions: "You are a concise product writer.",
  input: "Write a three-paragraph brief on async AI jobs.",
  background: true,
  store: true,
});

console.log(started.status);  // queued or in_progress — not final yet

let resp = started;
while (resp.status === "queued" || resp.status === "in_progress") {
  await sleep(2000);
  resp = await client.responses.retrieve(resp.id);
}

console.log(resp.status, resp.output_text);
Status What to do
queued Keep polling
in_progress Keep polling
completed Read output_text (or output items)
failed / cancelled Terminal — surface error or abort to user

Cancel in flight

await client.responses.cancel(responseId);
// Idempotent — calling cancel twice returns the final Response.

When to use it

Good fit Poor fit
Long-form generation (reports, briefs, codegen) Chat where user waits < 5s for first token — use streaming
Server-side jobs triggered from a webhook or queue Zero-data-retention compliance requirements
Notify user when done (email, push, inbox row) Sub-second interactive Q&A
Frontend mapping
Return response.id to the client immediately (202 Accepted pattern). Show a job status UI; poll your own API which calls responses.retrieve, or push via WebSocket when complete. Never block the browser on a minutes-long HTTP request.

Background + stream (optional)

You can set background: true and stream: true together — the job keeps running if the client disconnects, and you can resume with starting_after + the last sequence_number (streaming events reference). Useful for long streams on flaky mobile networks.
