Built-in Tools

Same tools array. Different owner of the handler.

In Lesson 7 you declared functions and ran them yourself — a two-turn loop with function_call_output. Built-in tools use the same request shape, but OpenAI's platform executes them. You get the answer in one response — no handler code, no second API call. Docs.

Custom vs built-in

Custom function Built-in tool
Runs on Your server OpenAI platform
Your code Handler + second create Declare tool in tools only
Output item function_call web_search_call, etc.
Best for Your DB, auth, business rules Web, files, code sandbox

Product rule of thumb: if the capability is generic and OpenAI already ships it, use built-in. If it touches your systems, use a custom function. See built-in tools reference.

Web search — one request

The most common built-in for product features that need current information. Add { type: "web_search" } to tools. The model decides whether to search based on the prompt. 

const response = await client.responses.create({
  model: "gpt-5",
  instructions: "Answer concisely. Cite sources when you search.",
  input: "What is the current stable version of the OpenAI Node SDK?",
  tools: [{ type: "web_search", search_context_size: "low" }],
});

console.log(response.output_text);
// "The current stable version is …" with inline citations
Model and cost
Web search requires a model that supports it (e.g. gpt-5 or gpt-4.1-mini) — not gpt-4.1-nano from earlier lessons. Each search action has per-tool pricing on top of tokens. Use search_context_size: "low" for quick lookups; see pricing.

What's in the response

Unlike custom functions, you don't send results back. Inspect response.output for two item types:

Item Contains
web_search_call Search status, action type (search), queries used
message output_text plus url_citation annotations 
const search = response.output.find(i => i.type === "web_search_call");
const msg = response.output.find(i => i.type === "message");

search?.action;           // { type: "search", query: "…" }
response.output_text;     // user-facing answer
msg?.content[0].annotations;  // [{ type: "url_citation", url, title, … }]
Frontend mapping
Show a "Searching…" state while the request is in flight (built-ins can take longer than a plain completion).
Render url_citation links visibly — OpenAI requires inline citations to be clear and clickable when you display web results to users.

Other built-ins (survey)

type Use when

code_interpreter Model needs to run Python (charts, data transforms) in a sandbox

file_search needs a vector store id; we'll practice that when you ship a doc-Q&A feature. For now, web search is enough to internalize the built-in pattern.

Mixing custom and built-in

You can put both in the same tools array — e.g. web search for public docs plus get_feature_status for your backlog. The model picks which to call. Custom calls still need your two-turn loop; built-ins complete inside the same response (or chain via previous_response_id if you continue the thread).
