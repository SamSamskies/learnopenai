Agents SDK

You learned the loop by hand. The SDK runs it for you.

Lesson 17 closed the four tool patterns. Lesson 15 wired the capstone yourself: stream turn 1 → run handler → stream turn 2. That two-turn loop is the right way to learn the platform. When orchestration grows — more tools, more turns, specialist agents — the Agents SDK takes over the loop while you keep the product logic. Agents guide.

Responses API vs Agents SDK

Responses API (Lessons 1–17) Agents SDK (this lesson)
You call responses.create each turn You call run(agent, input) once
You detect tool calls and loop SDK executes tools and loops until done
You chain previous_response_id SDK manages history, sessions, or server state
Full control over streaming events Optional stream: true + built-in traces
Rule of thumb
Stay on raw Responses API when you need tight control (custom UI reducer from Lesson 16, one-shot built-ins from Lesson 8). Reach for the SDK when the agent loop itself becomes your maintenance burden.

See Agents SDK reference and capstone pattern.

Three building blocks

Primitive Role
Agent Name, instructions, model, tools, optional outputType
tool() Zod params + execute — replaces your handler wiring
run() Starts the loop; returns finalOutput, newItems, history 
import { Agent, run, tool } from "@openai/agents";
import { z } from "zod";

const getFeatureStatus = tool({
  name: "get_feature_status",
  description: "Look up a product feature by slug.",
  parameters: z.object({ feature_slug: z.string() }),
  execute: async ({ feature_slug }) => {
    // same handler body as Lesson 15
    return JSON.stringify({ found: true, title: "Dark mode", ... });
  },
});

const TriageCard = z.object({
  feature_title: z.string(),
  current_status: z.enum(["shipped", "planned", "backlog", "unknown"]),
  priority: z.enum(["low", "medium", "high", "unknown"]),
  recommendation: z.string(),
});

const agent = new Agent({
  name: "Product Triage",
  model: "gpt-4.1-nano",
  instructions: "Use get_feature_status for backlog data.",
  tools: [getFeatureStatus],
  outputType: TriageCard,   // structured output — like zodTextFormat
});

const result = await run(agent, userQuestion, { maxTurns: 5 });
result.finalOutput;  // → typed TriageCard

The agent loop

Under the hood, run() repeats what you wrote manually in the capstone: 

1. Call model with instructions + tools + history
2. Model returns final text / structured output → done
3. Model returns tool calls → execute your execute() handlers
4. Append tool results → go to 1
5. Model requests handoff → switch agent → go to 1

maxTurns (default 10) is your safety rail — same reason you would cap a hand-rolled loop.

Capstone refactor

Lesson 15 (hand-rolled) Lesson 18 (Agents SDK)
Tool definition zodResponsesFunction + separate handler fn tool({ execute }) — schema + handler together
Structured output Turn 2 zodTextFormat outputType: TriageCard on the agent
Loop Two explicit responses.stream calls One run() — SDK loops as needed
Streaming UI You map events → phases (Lesson 16) run(agent, input, { stream: true }) or keep raw API for UI
Observability Your logs Auto traces in OpenAI dashboard
Streaming tradeoff
The capstone UI reducer (Lesson 16) maps raw Responses API events to phases. The SDK can stream run events, but many production apps keep the server on raw responses.stream for UI control and use the SDK for backend orchestration jobs. Both patterns are valid.

Handoffs — when one agent isn't enough

Define specialist agents and let a triage agent route between them: 

const backlogAgent = new Agent({
  name: "Backlog",
  instructions: "Answer roadmap questions using get_feature_status.",
  tools: [getFeatureStatus],
});

const docsAgent = new Agent({
  name: "Docs",
  instructions: "Answer from uploaded PRDs.",
  tools: [fileSearchTool(process.env.FILE_SEARCH_VECTOR_STORE_ID)],
});

const triage = Agent.create({
  name: "Triage",
  instructions: "Route backlog questions vs doc questions.",
  handoffs: [backlogAgent, docsAgent],
});

const result = await run(triage, "What does the PRD say about dark mode?");
console.log(result.lastAgent.name);  // → "Docs"

This is the natural next step after Lesson 17 — mix custom tools and file_search behind one entry point.

What the SDK adds beyond the loop

  • Traces — every run appears in the OpenAI dashboard for debugging
  • Sessions — persistent chat without manually passing history
  • Guardrails — block bad inputs/outputs before they reach users
  • Human-in-the-loop — needsApproval on tools pauses for review
  • Hosted tool helpers — webSearchTool(), fileSearchTool(), etc.
