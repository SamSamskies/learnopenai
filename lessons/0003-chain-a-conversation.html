Lesson 3 · ~15 minutes

Chain a Conversation

A chat product isn't one question. It's a thread — and the API needs to remember what came before.

In Lesson 1 you made a single call. In Lesson 2 you watched text arrive live. Real products need multi-turn state: the model must know what the user said two messages ago without you re-pasting the whole transcript every time.

The Responses API gives you three ways to carry context. This lesson teaches the simplest — previous_response_id — because it's what most chat UIs start with.

The old way (Chat Completions mental model)

If you've seen Chat Completions tutorials, multi-turn looked like this: grow a messages[] array, append the assistant reply, append the next user message, send the entire array on every request. 

messages.push(assistantMessage);
messages.push({ role: "user", content: "And its population?" });
// send ALL of messages again

That still works in Responses (pass prior output items back into input), but it's your job to manage the growing transcript. For most chat products there's a cleaner option.

Chain with previous_response_id

Save the resp_… id from each response. On the next turn, pass only the new user message — OpenAI replays prior context server-side. Docs. 

const first = await client.responses.create({
  model: "gpt-4.1-nano",
  input: "My favorite color is teal. Remember this.",
  store: true,
});

const second = await client.responses.create({
  model: "gpt-4.1-nano",
  input: "What is my favorite color?",
  previous_response_id: first.id,  // <— chains the thread
  store: true,
});

Turn 2's input is just the follow-up question — not the full history. The model should answer "teal" because it can see turn 1 through the chain.

Frontend mapping
Keep lastResponseId in React state (or your DB per thread). Each send: POST with input: userMessage and previous_response_id: lastResponseId. On success: setLastResponseId(response.id). "New chat" = clear the id and omit previous_response_id.

Three state strategies (pick one per product)

Approach You send each turn When to use
previous_response_id New user message only Default for chat threads
Manual item replay Prior output + new message Trimming context, full control, store: false
Conversations API Messages on a conversation object Threads that survive across sessions/devices

Keep the conversation state reference handy — you'll choose between these as products grow.

Gotchas that bite product engineers

Detail Why it matters
store: true Default. Server must store responses to resolve previous_response_id. Use store: false only when you replay context yourself.
instructions Top-level system prompts do not carry over the chain. Resend stable instructions on every turn.
Token billing Prior turns still count as input tokens each request. Chaining saves you from managing arrays — not from economics.
Where the id comes from Blocking call: response.id. Streaming: event.response.id on response.completed (Lesson 2).
Not free context
Even with previous_response_id, reconstructed history is billed as input. Repeated prefixes may hit prompt caching discounts — a later lesson topic.

Do this now

10-minute win: prove the model remembers across turns.

  1. Run: node practice/0003-chain.mjs
  2. Confirm turn 2 answers with "teal" (or similar)
  3. Compare usage.total_tokens — turn 2 should be higher than turn 1
  4. Change turn 2's question to something that only makes sense after turn 1
  5. Break it on purpose: omit previous_response_id and see the model forget

When turn 2 recalls turn 1 without you resending history, you have the state primitive every chat backend needs. Lesson 4 will add instructions and structured control over model behavior.
