Lesson 4 · ~15 minutes

Set System Instructions

Users type the question. You write the rules the model must follow.

In Lesson 3 you chained turns with previous_response_id. The model remembered what the user said — but who sets the product's behavior? Tone, format, guardrails, domain knowledge. That's system instructions.

In Chat Completions you tucked this in a system message. In Responses it's a top-level instructions parameter — separate from input, separate from the user's words. Docs.

instructions vs input

Parameter Who "writes" it Typical content
instructions Your product (developer) Role, tone, format rules, business constraints
input End user The actual question or task for this turn 
const response = await client.responses.create({
  model: "gpt-4.1-nano",
  instructions: "You are a terse assistant. Never use more than five words.",
  input: "What color is the sky?",
});

instructions takes priority over conflicting content in input. If the user asks for a paragraph but your instructions say "five words max," the instructions win.

The chaining gotcha (you already hit this)

previous_response_id replays stored user/assistant turns — it does not replay instructions from earlier requests. You must resend stable instructions on every turn. 

// Turn 2 — BOTH are required for a chained chat product
const second = await client.responses.create({
  model: "gpt-4.1-nano",
  instructions: sameInstructionsAsTurn1,  // <— resend every time
  input: "What about at sunset?",
  previous_response_id: first.id,
  store: true,
});
Common bug
Chain works, memory works, but the bot "forgets" its personality on turn 2. Cause: instructions sent only on turn 1. Fix: include them on every call.
Frontend mapping
Store instructions in a prompt module (or DB config per feature) — not in React state per message. On each send: { instructions, input: userMessage, previous_response_id }. "New chat" clears the response id; instructions stay the same.

instructions vs developer role

You can also put system guidance in input as a developer role message. That does persist in the stored transcript when chaining. Most products still prefer top-level instructions because you can change behavior per request without rewriting history.

Approach Persists when chaining? Best for
instructions No — resend each turn Default: versioned product prompts
developer in input Yes — part of transcript Rules baked into thread history (uncommon)

See the instructions reference and conversation state reference together — chaining and instructions are a pair.

Version prompts like code

Treat prompts as application logic, not copy-pasted strings. OpenAI recommends keeping prompt builders in a small module, pinning model snapshots in production, and testing prompt changes before rollout. 

// prompts/support-bot.mjs
export function buildSupportInstructions({ companyName }) {
  return `
You are a support assistant for ${companyName}.
Answer in one short paragraph. Never invent pricing.
If unsure, say you will escalate.
  `.trim();
}

Do this now

10-minute win: prove instructions shape behavior — and vanish when you forget to resend them.

  1. Run: node practice/0004-instructions.mjs
  2. Turns 1–2 should stay under six words (terse rule)
  3. Turn 3 should ramble — you chained but omitted instructions
  4. Fix turn 3: add instructions back and rerun that call
  5. Extract the terse prompt into a const or small module — that's the product pattern

When you can resend stable instructions across a chained thread, you have the two primitives every chat backend needs: state + behavior. Lesson 5 will constrain output with structured JSON so your UI can parse responses reliably.
