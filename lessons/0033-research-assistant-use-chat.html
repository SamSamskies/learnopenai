Lesson 33 · ~40 minutes

useChat Client Migration

You learned the wire by hand. Now let the SDK own the chat loop.

Lesson 32 moved the server to streamText but deliberately kept your custom SSE frames so the client and Vitest stayed stable. This lesson closes the loop: migrate ResearchChat.tsx to useChat and switch the route to the AI SDK UI Message Stream protocol — with a typed data-research part that carries your existing ResearchUIState snapshots.

What changes (and what doesn’t)

Unchanged Migrated
lib/stream-research.ts phase machine Route: custom SSE → createUIMessageStreamResponse
BriefArticle, guards, sessions, upload Client: fetchEventSource → useChat
ResearchUIState shape Wire: raw JSON snapshots → data-research parts
previousResponseId via session store Busy/abort: manual useState → status + stop()
Vitest route contract (phase order, final brief) Test helper parses UI message stream, not raw ResearchUIState

Architecture after migration

ResearchChat (useChat)          route.ts                    stream-research.ts
  │                                  │                              │
  │── POST { messages, sessionId } ─►│ createUIMessageStream        │
  │                                  │   writer.write(data-research)│
  │                                  │   onSnapshot → same state ───►│ streamText loop
  │◄── UI message stream (SSE) ─────│◄─────────────────────────────│
  │    message.parts[].data-research │                              │
  │    status / stop / error         │                              │

The client no longer parses data: { phase, … } itself. useChat owns message history, streaming status, and abort. Your product UI still reads the same ResearchUIState — it just lives inside a reconciled data-research part on the assistant message.

Step 2 — Type-safe UI messages

Create lib/research-ui-message.ts. Attach your snapshot shape as a custom data part: 

import type { UIMessage } from "ai";
import type { ResearchUIState } from "@/lib/research-state";

export type ResearchUIMessage = UIMessage<
  never,
  { research: ResearchUIState }
>;

export function getResearchPart(message: ResearchUIMessage) {
  return message.parts.find((p) => p.type === "data-research");
}

The data-research prefix comes from the key in the generic (research → data-research).

Step 3 — Route: UI message stream

Replace the hand-rolled ReadableStream in app/api/research/route.ts with createUIMessageStream + createUIMessageStreamResponse. Extract the latest user text from messages; keep sessionId from the request body for threading and file search. 

import {
  createUIMessageStream,
  createUIMessageStreamResponse,
  type UIMessage,
} from "ai";

const RESEARCH_PART_ID = "research-1";

export async function POST(req: Request) {
  // … guards …
  const { messages, sessionId } = await req.json();
  const message = lastUserText(messages); // helper: find latest user text part

  const stream = createUIMessageStream<ResearchUIMessage>({
    execute: async ({ writer }) => {
      try {
        const previousResponseId = getLastResponseId(sessionId) ?? undefined;
        const { vectorStoreId } = getSession(sessionId);

        const responseId = await streamResearch(message, (state) => {
          writer.write({
            type: "data-research",
            id: RESEARCH_PART_ID,
            data: state,
          });
        }, { previousResponseId, vectorStoreId });

        if (responseId) setLastResponseId(sessionId, responseId);
      } catch (err) {
        writer.write({
          type: "data-research",
          id: RESEARCH_PART_ID,
          data: createResearchState({
            phase: "error",
            error: publicError(err),
          }),
        });
      }
    },
  });

  return createUIMessageStreamResponse({ stream });
}

Reconciliation: reuse the same id on every writer.write call. The client updates one part in place instead of appending a new snapshot per frame — same UX as setState(snapshot) on every SSE frame.

Step 4 — Client: useChat

Replace fetchEventSource, turns, pendingQuestion, and manual state with useChat: 

import { useChat } from "@ai-sdk/react";
import { DefaultChatTransport } from "ai";

const { messages, sendMessage, status, stop, setMessages, error } =
  useChat<ResearchUIMessage>({
    transport: new DefaultChatTransport({
      api: "/api/research",
      headers: () => authHeaders(),
      body: () => ({ sessionId }),
    }),
  });

const busy = status === "submitted" || status === "streaming";

Send on submit:

sendMessage({ text: message });

Render history from messages:

  • User turns — message.role === "user", text from part.type === "text"
  • Assistant turns — getResearchPart(message)?.data drives LiveTurn / BriefArticle
  • In-flight — last assistant message with phase !== "done"
  • Errors — error from hook, or phase === "error" on the data part

newChat: call setMessages([]) after clearing the session server-side. Optional: wire stop() to a Stop button while status === "streaming".

Step 5 — Update Vitest

The response is still SSE, but frames follow the UI message stream protocol — not bare ResearchUIState JSON. Update lib/read-sse-stream.ts (or add a sibling helper) to track data-research reconciliation events and extract the latest ResearchUIState.

Keep the same three assertions from Lesson 31:

  1. 400 when message / user text is missing
  2. 401 when secret is wrong
  3. Phase order: searching → streaming-answer → done, with brief + sources

The route now reads messages from the body — send a minimal UIMessage[] with one user text part in the test request.
