What the server remembers

OpenAI stores each response when store: true. Your server stores one id per chat session — the final response id from the previous user message (conversation state reference).

Path Responses in one user message Save this id
Backlog (custom tool) Turn 1: function_call · Turn 2: structured card Turn 2 response.id
Docs (file_search) Single response with answer + citations That response's id
Direct (no tool) Model answers from prior context That response's id 
// Per session on your server:
sessions.set(sessionId, lastResponseId);

// Each POST /api/triage:
await streamTriageDocs(onSnapshot, message, vectorStoreId, {
  previousResponseId: sessions.get(sessionId),
});
// → return lastResponseId from final response.completed / turn 2
Resend every turn
instructions and tools do not carry over the chain. The library still sends them on turn 1; your route must keep doing so. Only the user message and previous_response_id are new each turn.

Session boundary

A session is your product's chat thread — not OpenAI's Conversations API (that comes later for cross-device persistence). For this lesson:

  • Client generates a sessionId (UUID in sessionStorage)
  • Server maps sessionId → lastResponseId in memory
  • New chat clears the map entry and issues a fresh UUID

In production you persist lastResponseId in your DB next to the thread row — same pattern, durable storage.
