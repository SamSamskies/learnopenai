Ship the Triage API

Terminal proved the loop. Now a user sees it.

Lesson 15 built the two-turn flow. Lesson 16 mapped it to UI phases. Lesson 18 showed when the SDK can run the loop for you. Your mission success criteria still need the missing piece: a shipped feature — API key on the server, streaming updates to a browser.

The server boundary

The browser never calls OpenAI directly. Your server owns:

  1. OPENAI_API_KEY (from .env on the server)
  2. The two-turn capstone loop (stream tool → handler → stream card)
  3. Mapping raw API events → compact UI state snapshots

See streaming UI state reference and capstone pattern.

What crosses the wire

Do not forward raw OpenAI stream events to the browser. The server maps them to the same shape from Lesson 16: 

// Each SSE message — one UI snapshot
{
  "phase": "streaming-args",
  "toolName": "get_feature_status",
  "argsPreview": "{\"feature_slug\":\"dark-mode\"}",
  "cardPreview": "",
  "card": null,
  "error": null
}

// Final message
{
  "phase": "done",
  "card": {
    "feature_title": "Dark mode",
    "current_status": "planned",
    "priority": "high",
    "recommendation": "..."
  }
}

The client switches on phase — identical logic to practice/0016-ui-state-machine.mjs, but rendered in HTML instead of [UI] log lines.

Raw API vs Agents SDK
Use the hand-rolled two-turn loop here — you control every SSE snapshot for the UI. Use the Agents SDK from Lesson 18 for backend jobs where traces matter more than per-event UI mapping.
