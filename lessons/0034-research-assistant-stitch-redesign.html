Redesign Research Assistant in Stitch

One screen. Many states. Design them all before you code.

Lesson 33 finished the AI SDK migration — the app works, but it still looks like a prototype. This lesson uses Google Stitch to redesign the entire Research Assistant surface in one pass. Because the app is a single page, you can treat it as one Stitch project with multiple artboards for each UI phase.

Related: Google Stitch reference · Design to code reference · Research Assistant reference · Lesson 33 (current UI). Next: Lesson 35 (implement DESIGN.md in code).

What you will produce

Artifact Where it lives
High-fidelity mockups for every UI phase Stitch canvas at stitch.withgoogle.com
DESIGN.md — colors, type, spacing, components research-assistant/DESIGN.md (export from Stitch)

You are not writing React in this lesson. Lesson 35 applies the design system to ResearchChat.tsx.

Step 0 — Inventory the one screen

ResearchChat.tsx is the whole product UI. It has persistent chrome and transient states driven by ResearchUIState.phase. Map these before you prompt:

Region Always visible? Notes
Header — title, Upload, New chat Yes Upload accepts .pdf, .md, .txt
Uploaded file chips When files indexed Show filename badges above thread
Empty state Before first message Invite a research question; no thread yet
User bubble Per turn “You” label + question text
Searching phase === "searching" “Searching the web…” / docs / both
Brief preview streaming-answer Collapsible raw JSON while streaming
Research brief card phase === "done" Headline, summary, key points, sources, confidence
Error phase === "error" Human-readable failure message
Composer Yes Auto-growing textarea + send button; pinned bottom

Brief schema fields (for the done-state card): headline, summary, key_points[], confidence (low | medium | high), plus a Sources list (URLs and file names) and grounding badges (“Grounded with web search”, etc.).

Step 1 — Open Stitch and start a project

  1. Go to stitch.withgoogle.com and sign in with your Google account.
  2. Create a new project. Name it Research Assistant.
  3. In model settings, prefer Gemini 3.1 for the first pass (better layout interpretation than Flash). You can remix with Nano Banana later for bolder visuals.
  4. Optional: run npm run dev in research-assistant/, screenshot the current page, and drop it on the canvas as reference — not as the target look.

Step 2 — Paste the starter prompt

Copy this prompt into Stitch’s project description or first generation request. It asks for five screens — empty, searching, streaming, done, and error — on one cohesive design system. 

Design a single-page web app called "Research Assistant" for solo builders who research topics with AI.

Platform: desktop web, max content width ~680px, centered on a light neutral page background. Support light mode first; subtle dark mode variant optional.

Brand feel: calm, editorial, trustworthy — like a sharp research brief, not a generic chatbot. Avoid purple gradients and stock "AI sparkle" clichés.

Persistent chrome (every screen):
- Header row: app title "Research Assistant" on the left; "Upload" and "New chat" text buttons on the right (Upload indexes PDF/MD/TXT for grounded answers).
- Bottom composer: rounded input bar with placeholder "Ask a research question…", auto-growing textarea feel, circular send button on the right. Composer stays fixed at bottom.

Screen 1 — Empty state:
- No conversation yet. Short welcome line inviting a research question.
- Composer enabled.

Screen 2 — Searching:
- User bubble: label "You", question "What changed in Next.js App Router in 2025?"
- Status line: "Searching the web…" with subtle progress indicator.
- Composer disabled while working.

Screen 3 — Streaming answer:
- Same user bubble.
- Collapsible "Brief preview" panel showing monospace JSON being filled in (structured brief streaming).
- Composer disabled.

Screen 4 — Done (research brief card):
- User bubble + a prominent brief card with:
  - headline (large)
  - summary paragraph (2–3 sentences)
  - bulleted key_points (3–5 items)
  - Sources section: linked titles for web URLs + file icons for uploaded docs
  - footer meta row: "Grounded with web search", "Grounded with documents", "Confidence: high"
- Show a second older turn above (faded) to imply multi-turn thread.
- Composer enabled.

Screen 5 — Error:
- User bubble + inline error message in accessible red tone.
- Composer enabled so user can retry.

Typography: clear hierarchy (display title, section labels, body, meta captions). Generous whitespace. Rounded cards with subtle borders, not heavy shadows.

Stitch may generate up to five starter screens in one shot. Review all of them on the infinite canvas before iterating.

Step 3 — Iterate with targeted follow-ups

Vibe design is conversational. Use short, specific prompts per screen — don’t regenerate the whole project unless the direction is wrong.

Goal Example follow-up prompt
Polish the brief card “Make the research brief card feel more editorial: stronger headline, summary in slightly larger body text, key points as a clean numbered list, sources in a compact link list with external-link icons.”
Improve empty state “Replace generic welcome copy with one line: ‘Ask anything — I’ll search the web and your uploaded docs.’ Add three subtle example question chips users can click.”
Clarify streaming “In the streaming screen, show the brief preview panel half-open with a pulsing cursor in the JSON — make it obvious this is interim, not the final brief.”
Upload affordance “Add a row of small file chips under the header when documents are indexed: pill badges with file icon + filename.”
Explore direction “Show me three color palette variants: warm paper, cool slate, and high-contrast ink.”

Tip: Stitch’s design agent can reason across the whole canvas. Point at a specific screen when you critique — “on the Done screen, increase contrast on source links.”

Step 4 — Stitch screens into a flow (optional)

Use Stitch’s Play mode to link screens: Empty → Searching → Streaming → Done. Click-through validates that transitions feel natural before you commit to code. You only have one route ( app/page.tsx) — the flow is logical, not navigational.

Step 5 — Export DESIGN.md

  1. When you are happy with the design system (not every pixel — the system matters), open Stitch export options.
  2. Select DESIGN.md. Stitch extracts colors (hex), typography, spacing scale, and component patterns from your canvas.
  3. Save the file to research-assistant/DESIGN.md in this repo.
  4. Skim the export. Add any rules Stitch missed — e.g. “brief card always uses rounded-xl”, “source links open in new tab”.

DESIGN.md is the bridge to Cursor. In Lesson 35 you will ask the agent to restyle ResearchChat.tsx against this file while keeping all behavior identical.

Step 6 — Sanity check before you leave Stitch

  • All five phases are represented and visually distinct
  • Header + composer are consistent across screens
  • Brief card shows every field: headline, summary, key points, sources, confidence, grounding badges
  • Error and disabled states are obvious
  • DESIGN.md is committed (or ready to commit) in research-assistant/
