Lesson 26 · ~30 minutes

Research API

Prove the server loop in the terminal. Wire the browser later.

Lesson 25 gave you the shell. Before touching React state, add a route handler that calls OpenAI with web_search and returns JSON. Same discipline as Lesson 19 — server owns the API key; you test with curl first.

Why API before UI

Streaming UI (Lesson 27) is harder to debug than a single POST. If citations are wrong or the model skips search, you want to see that in a JSON body — not buried in React state. One vertical slice: request in, answer + sources out.

Architecture

curl / Postman              Next.js route              OpenAI
     │  POST /api/research              │
     │  { "message": "..." }            │
     │─────────────────────────────────►│ openai.responses.create
     │                                  │ tools: [{ type: "web_search" }]
     │◄── { answer, citations } ────────│ platform runs search → message
     │                                  │

web_search is a built-in tool from Lesson 8 — no handler, no second turn. Compare with practice/0008-web-search.mjs.

1. OpenAI client singleton

Create lib/openai.ts so every route imports the same client. Next.js loads .env.local automatically — the key never reaches the browser. 

// lib/openai.ts
import OpenAI from "openai";

export const openai = new OpenAI();

2. Route handler

App Router maps app/api/research/route.ts → POST /api/research. Route Handlers docs. 

// app/api/research/route.ts
import { NextResponse } from "next/server";
import { openai } from "@/lib/openai";

type Citation = { title: string; url: string };

export async function POST(req: Request) {
  const body = await req.json();
  const message = body?.message;

  if (typeof message !== "string" || !message.trim()) {
    return NextResponse.json(
      { error: "message is required" },
      { status: 400 }
    );
  }

  const response = await openai.responses.create({
    model: "gpt-5-mini",
    instructions:
      "You are a research assistant for solo builders. Answer clearly and concisely. Cite sources when you use web search.",
    input: message.trim(),
    tools: [{ type: "web_search", search_context_size: "low" }],
  });

  const outputMessage = response.output.find((item) => item.type === "message");
  const annotations = outputMessage?.content?.[0]?.annotations ?? [];

  const citations: Citation[] = annotations
    .filter((note) => note.type === "url_citation")
    .map((note) => ({
      title: note.title ?? note.url,
      url: note.url,
    }));

  return NextResponse.json({
    answer: response.output_text,
    citations,
    searched: response.output.some((item) => item.type === "web_search_call"),
  });
}
Citations come from annotations, not the prose.
The model may hallucinate inline URLs. For UI links, use message.content[0].annotations where type === "url_citation" — same rule as practice/0008-web-search.mjs.

3. Response shape

{
  "answer": "Next.js 15 introduced ...",
  "citations": [
    { "title": "Next.js 15 Blog", "url": "https://..." }
  ],
  "searched": true
}

searched is a convenience flag — useful when the model answers from training data without calling web_search.

4. Test with curl

With npm run dev running:

curl -s http://localhost:3000/api/research \
  -H "Content-Type: application/json" \
  -d '{"message":"What changed in Next.js 15 App Router? Two sentences."}' \
  | jq

Confirm:

  • answer is non-empty prose
  • citations has at least one entry with a real URL
  • searched is true for a current-events question
