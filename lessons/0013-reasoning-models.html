Reasoning Models

Lesson 12 let jobs run for minutes. Reasoning models spend invisible tokens before you see the first word.

gpt-4.1-nano answers in one pass (Lesson 1). Reasoning models (e.g. o4-mini, gpt-5.4-mini, gpt-5.5) think first — planning, checking alternatives, recovering from ambiguity — then write the visible answer. Same Responses API; add reasoning: { effort: "…" }. Reasoning models guide.

What you pay for

Token type Visible? Billed as
Input Your prompt Input tokens
Reasoning No — hidden "thinking" Output tokens
Output output_text Output tokens

Read the bill in usage.output_tokens_details.reasoning_tokens. A short answer can still cost many tokens if the model thought hard. This is separate from prompt caching — caching discounts repeated input prefixes; reasoning is fresh output every call.

Reasoning effort

reasoning.effort is your latency/cost dial. Supported values depend on the model (low, medium, high, and on newer models none, minimal, xhigh).

Effort Reach for it when…
low Default experiment — tool use, support chat, drafting
medium Agentic coding, research, tasks needing judgement
high / xhigh Hard debugging or deep planning — only when evals prove the gap 
const response = await client.responses.create({
  model: "o4-mini",
  reasoning: { effort: "low" },
  input: "A bat and ball cost $1.10 total. The bat costs $1.00 more than the ball. How much is the ball?",
  store: true,
});

console.log(response.output_text);
console.log(response.usage.output_tokens_details.reasoning_tokens);
Product rule
Start every new feature on the cheapest model that passes your eval. Add reasoning models only for tasks that fail that bar — then tune effort up, not down, based on measured quality gaps.

Reasoning summaries (optional)

Raw reasoning tokens are not exposed. To show users that the model thought (without the full chain-of-thought), request a summary: 

reasoning: { effort: "low", summary: "auto" }
// Inspect response.output for type === "reasoning" → summary[].text

Useful for "show your work" UIs in coding or research products. Requires org verification on some models — see the docs.

Tool loops + reasoning

When a reasoning model calls your custom tools (Lesson 10), pass reasoning items forward on the next turn — easiest via previous_response_id. Dropping them wastes the model's prior thinking and can hurt quality.

Stateless / zero-data-retention: add include: ["reasoning.encrypted_content"] so encrypted reasoning blobs round-trip without store: true. See reasoning models reference.

Don't run out of budget

max_output_tokens caps all generated tokens — reasoning and visible text. Set it too low and you get status: "incomplete" with incomplete_details.reason === "max_output_tokens" — sometimes with no output_text at all. Leave headroom when experimenting; pair long high-effort jobs with background mode.

When not to use reasoning

Good fit Poor fit
Multi-step debugging, codegen, data validation Classification, extraction, simple Q&A on nano
Agentic tool loops needing planning Latency-sensitive voice / sub-second chat
Tasks where evals show nano/mini fail "Just in case" — reasoning is a cost multiplier
