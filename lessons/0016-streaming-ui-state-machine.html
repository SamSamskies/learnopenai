Streaming UI State Machine

The API streams events. The UI streams phases.

Lesson 15 ran the full triage flow in the terminal. Your mission success criteria still need the other half: a streaming UI a user actually sees. This lesson maps API events to UI state — the pattern behind every chat surface, form filler, and agent panel you ship.

Events are not UI state

A common mistake: storing raw stream events in React state and rendering from them. Events are inputs to a reducer; the UI reads phases and derived fields.

Layer Owns
API (server) Two-turn loop, tool handler, previous_response_id
Transport SSE or fetch stream — forward mapped UI updates to the browser
UI (client) Phase machine, previews, typed card at the end

See streaming UI state reference and capstone pattern.

The phase machine

For the capstone triage feature, six phases cover the full user journey: 

idle → streaming-args → running-handler → streaming-card → done
                                                              ↘ error
Phase User sees Triggered by
idle Input ready, send button enabled Initial / after reset
streaming-args “Calling get_feature_status…” + live args JSON output_item.added (function_call)
running-handler Spinner — “Looking up backlog…” Turn 1 complete; your handler runs
streaming-card Skeleton card or raw JSON preview Turn 2 stream starts
done Typed TriageCard — title, status, priority, recommendation finalResponse().output_parsed
error Error message + retry Handler throw, API error, Zod parse failure
Still don't parse partial JSON
During streaming-card, show a skeleton or monospace preview. Render typed fields only when phase === "done" — same rule as Lesson 6.

Reducer sketch (React)

Your frontend experience maps cleanly to useReducer. The server sends compact UI events; the reducer updates phase and previews: 

type Action =
  | { type: "SUBMIT" }
  | { type: "TOOL_STARTED"; toolName: string }
  | { type: "ARGS_DELTA"; delta: string }
  | { type: "ARGS_DONE"; args: string }
  | { type: "HANDLER_START" }
  | { type: "CARD_STREAM_START" }
  | { type: "CARD_DELTA"; delta: string }
  | { type: "CARD_DONE"; card: TriageCard }
  | { type: "ERROR"; message: string };

function triageReducer(state, action) {
  switch (action.type) {
    case "SUBMIT":
      return { ...initialState, phase: "streaming-args" };
    case "TOOL_STARTED":
      return { ...state, phase: "streaming-args", toolName: action.toolName };
    case "HANDLER_START":
      return { ...state, phase: "running-handler" };
    case "CARD_STREAM_START":
      return { ...state, phase: "streaming-card", cardPreview: "" };
    case "CARD_DONE":
      return { ...state, phase: "done", card: action.card };
  }
}

Server boundary

The browser never holds your API key. An API route runs the capstone loop and streams mapped events to the client: 

// app/api/triage/route.ts (sketch)
export async function POST(req: Request) {
  const { question } = await req.json();
  const stream = new ReadableStream({
    async start(controller) {
      const send = (payload) =>
        controller.enqueue(`data: ${JSON.stringify(payload)}\n\n`);

      send({ type: "TOOL_STARTED", toolName: "get_feature_status" });
      // … turn 1 events → send ARGS_DELTA, ARGS_DONE
      send({ type: "HANDLER_START" });
      // … run getFeatureStatus, start turn 2
      send({ type: "CARD_STREAM_START" });
      // … CARD_DELTA optional
      send({ type: "CARD_DONE", card: response.output_parsed });
      controller.close();
    },
  });
  return new Response(stream, {
    headers: { "Content-Type": "text/event-stream" },
  });
}
