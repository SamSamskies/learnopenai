Prompt Caching

Lesson 10 chained threads. Every turn still bills input tokens — but not all at full price.

You resend instructions every turn (Lesson 4). Chained history replays server-side (Lesson 10). That growing prefix is expensive — unless OpenAI has seen the exact same start recently. Prompt caching guide.

What it is (and isn't)

Detail
Automatic On by default for gpt-4o+. No flag to enable, no extra fee.
Prefix match Only the identical start of your prompt can hit cache.
Does not change output Same model, same prompt → same answer. Caching is economics + latency only.
Not conversation memory That's previous_response_id. Caching discounts token prefill.

The 1024-token floor

Caching activates when the prompt prefix is at least 1024 tokens. Shorter prompts still return cached_tokens: 0 — check the field anyway so your observability is ready when prompts grow. 

const cached =
  response.usage?.input_tokens_details?.cached_tokens ?? 0;
const input = response.usage?.input_tokens ?? 0;

console.log(`${cached} / ${input} input tokens were cache hits`);

On gpt-4.1-nano, cached input tokens bill at $0.025 / 1M vs $0.10 / 1M uncached — a 75% discount on the cached portion. Pricing.

Static front, dynamic end

Cache hits require an exact prefix. Structure every request so stable content comes first and per-user content comes last: 

await client.responses.create({
  model: "gpt-4.1-nano",
  instructions: productHandbook,   // same every request — FRONT
  tools,                            // stable tool defs — FRONT
  input: userMessage,               // changes each turn — END
  previous_response_id: lastId,     // chained history — middle/end
  store: true,
});
Put at the front (static) Put at the end (dynamic)
System instructions, brand voice, policies User's message this turn
Tool definitions (tools array) Session-specific variables
Few-shot examples, product FAQ Timestamps, user name, locale
Structured-output schema Per-request RAG snippets that differ
Cache breaker
instructions: `Today is ${new Date()}` in your system prompt changes every request — zero cache hits. Pass the date in input instead.

How chaining interacts

With previous_response_id, the server replays prior turns into the prefix. Turn 3's prefix is roughly: 

[instructions] + [tools] + [turn 1] + [turn 2] + [new user message]

You still resend identical instructions each turn — that block can cache even as the conversation tail grows. The new user message and latest turn are uncached; the shared front is discounted.

Frontend mapping
Store instructions in a versioned prompt module — not rebuilt per render. Log cached_tokens per request in your API layer. Alert when a deploy changes instructions and cache hit rate drops.

Optional knobs

Parameter When to use
prompt_cache_key Many tenants share the same long prefix — route them to the same cache shard.
prompt_cache_retention "24h" on supported models (e.g. gpt-4.1) for longer retention vs in-memory.

Skip these until you have cache metrics. The default behavior is enough for v1 products.
