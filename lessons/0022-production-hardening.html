Production Hardening

Demo proved the feature. Now protect the route.

Lesson 21 completed the capstone: backlog tool, file search, fetch client, SSE snapshots. A real product still needs a server boundary you can ship — auth before the stream opens, rate limits before OpenAI bills you, and errors that do not leak platform internals.

Two error channels

Streaming routes have two failure modes. Mixing them confuses the client and breaks React error boundaries.

Failure When Response
Pre-stream Auth, rate limit, invalid body — before SSE starts HTTP 401 / 429 + JSON { error }
Mid-stream OpenAI error after bytes already sent SSE snapshot { phase: "error", error: "…" }

See production hardening reference and fetch stream client. 

// Client — check HTTP status BEFORE parsing SSE
const res = await fetch("/api/triage", { headers: { Authorization: `Bearer ${token}` } });
if (!res.ok) {
  const { error } = await res.json();
  dispatch({ phase: "error", error });
  return;
}
await readSseStream(res, dispatch);
Never send OPENAI_API_KEY to the browser.
The Bearer token in this lesson is an app secret (TRIAGE_API_SECRET) — proof the caller may use your route. In production that becomes a session cookie or JWT from your auth provider.

Auth guard

Run checks before writeHead(200, { "Content-Type": "text/event-stream" }). Once you commit to SSE, the client expects frames — not a JSON error body. 

const auth = guard.checkAuth(req);
if (!auth.ok) {
  sendJsonError(res, auth.status, auth.error);  // 401
  return;
}

// Only now open the stream
res.writeHead(200, { "Content-Type": "text/event-stream", … });

Rate limiting

AI routes are expensive. A simple per-IP bucket (5 req/min in the practice script) stops accidental loops and basic abuse. Return 429 as pre-stream JSON — same as auth failures.

practice/lib/triage-guard.mjs holds auth + rate limit. Swap the in-memory map for Redis when you run multiple server instances.
