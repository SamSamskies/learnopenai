State Across Tool Turns

Lesson 3 taught chaining. Lesson 7 taught tools. A real chat thread needs both — and one id rule most people get wrong.

Your product assistant can look up features, stream progress, and answer follow-ups. But which response.id do you save after a tool loop? Chain on the wrong one and the next user message loses the tool result — or worse, replays a half-finished turn. Conversation state guide.

The id rule after a tool loop

A custom tool cycle is two API calls (Lesson 7): turn 1 ends with a function_call; turn 2 sends function_call_output and gets the user-facing answer.

Response Contains Save as lastResponseId?
Turn 1 (first.id) Tool call only — no final answer yet No
Turn 2 (second.id) Tool output + assistant reply Yes 
// After the tool loop completes:
lastResponseId = second.id;  // not first.id

// Next user message:
const next = await client.responses.create({
  model: "gpt-4.1-nano",
  instructions: INSTRUCTIONS,           // resend every turn
  input: "What about SSO?",
  previous_response_id: lastResponseId,
  tools,                                // still declare tools
  store: true,
});
Streaming
When stream: true, read event.response.id from response.completed on the final response in the loop — turn 2 after tools, or turn 1 when no tool was called (Lesson 9).

Multi-turn chat with tools

Lesson 3 chained plain text. With tools, the pattern is the same — only the id you save changes:

Per user message Your backend does
Send input + previous_response_id: lastResponseId + instructions + tools
Tool loop (if any) Run handler → function_call_output → get final response
On complete lastResponseId = finalResponse.id
New chat Clear lastResponseId; omit previous_response_id

The model can answer turn 3 from memory of turn 1's tool results — you do not resend function_call_output yourself. Server-side chaining replays the full thread.

Three state strategies (revisited)

You met these in Lesson 3. Now you can choose with product context:

Approach When to pick it
previous_response_id Default chat threads — including tool-assisted ones
Manual item replay store: false, trim old turns, or custom context windows
Conversations API Threads that survive across sessions, devices, or long retention

See the conversation state reference for the full comparison.

Gotchas that bite at scale

Detail Why it matters
instructions Do not carry over the chain. Resend stable system prompts every turn.
tools Declare on every turn that might need them — they are not inherited.
Token billing Prior turns (including tool calls and outputs) bill as input each request.
store: false Server cannot resolve previous_response_id — you must replay output items yourself.
Retention Stored responses expire after ~30 days unless attached to a Conversations object.
Frontend mapping
Store lastResponseId per thread in React state or your DB. After each completed loop (streamed or blocking), persist the final id. "New chat" clears it.
