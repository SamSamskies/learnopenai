Stream a Response

Users don't wait for the whole answer. They watch it arrive — word by word.

In Lesson 1 you waited for the full response object. That works for scripts, but chat products feel broken without streaming: text appearing incrementally while the model still generates.

Your mission includes shipping a streaming UI. This lesson teaches the server-side primitive — the same event loop every chat interface builds on.

Blocking vs streaming

Mode What you get When to use
Blocking (default) One Response object when generation finishes Batch jobs, short prompts, backend transforms
Streaming A sequence of events over SSE Chat UIs, anything the user watches live

Streaming doesn't make the model faster — it makes perceived latency lower. The first token often arrives in hundreds of milliseconds; the user reads while the rest generates. Docs.

Turn streaming on

Add one parameter to the call you already know:

const stream = await client.responses.create({
  model: "gpt-4.1-nano",
  input: "Explain why streaming feels faster in three short sentences.",
  stream: true,  // <— this changes everything
});

Instead of a Response, you get an async iterable of stream events. Loop with for await: 

for await (const event of stream) {
  console.log(event.type);
}

You'll see types like response.created, response.output_text.delta, and response.completed. Keep the streaming events reference open — you'll return to it often.

The event that matters for text

For a plain text reply, you mostly care about one event: response.output_text.delta. Each one carries a small string in event.delta. 

let text = "";

for await (const event of stream) {
  if (event.type === "response.output_text.delta") {
    process.stdout.write(event.delta); // terminal: print as it arrives
    text += event.delta;               // UI: same idea — append to state
  }
}
Frontend mapping
In React, each delta is setReply(prev => prev + event.delta). In a browser chat UI, your API route forwards SSE chunks to the client the same way. The append pattern is identical whether you're in Node or the DOM.

Know when you're done

response.completed fires once at the end. The full response object lives on event.response — same shape as Lesson 1's blocking call (id, usage, output_text). 

if (event.type === "response.completed") {
  console.log("id:", event.response.id);
  console.log("tokens:", event.response.usage?.total_tokens);
}

Save the resp_… id here if you need multi-turn state later. Your accumulated text string should match event.response.output_text.

What you'll ignore (for now)

Streams also emit tool-call deltas, reasoning summaries, and image events. Filter by event.type — only handle what your product needs. Tool streaming comes in a later lesson.

Errors
On failure you'll get response.failed instead of response.completed. Always handle both in production UIs.
