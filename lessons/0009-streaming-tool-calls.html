Lesson 9 · ~15 minutes

Streaming Tool Calls

Lesson 7 taught the loop. Lesson 2 taught the stream. Now wire them together for a live UI.

A blocking tool call leaves the user staring at a spinner until turn 1 finishes. Streaming lets you show progress: which tool the model picked, arguments filling in character by character, then the final answer token by token. Function calling docs · Streaming guide.

Same loop, streamed events

The two-turn pattern from Lesson 7 does not change: declare tools → model calls one → your handler runs → model answers. You add stream: true to each responses.create and handle new event types instead of waiting for one object.

Turn Events to handle UI shows
1 — tool call response.output_item.added, response.function_call_arguments.delta, response.function_call_arguments.done “Calling get_feature_status…” + live args preview
Between Your handler (blocking is fine — usually fast) “Running lookup…” spinner
2 — answer response.output_text.delta (Lesson 2) Final reply streaming in

Turn 1 — tool call events

When the model starts a function call, you get response.output_item.added with the tool name and call_id — show those immediately. 

if (event.type === "response.output_item.added"
    && event.item.type === "function_call") {
  toolName = event.item.name;
  callId = event.item.call_id;
  // UI: "Calling get_feature_status…"
}

Arguments arrive as JSON text chunks via response.function_call_arguments.delta. Append each event.delta to a buffer — same append pattern as text streaming, but you're building a JSON string. 

if (event.type === "response.function_call_arguments.delta") {
  argsBuffer += event.delta;
  // UI: show partial JSON or a "preparing request…" preview
}

if (event.type === "response.function_call_arguments.done") {
  argsBuffer = event.arguments; // final JSON string — safe to JSON.parse now
}
Don't parse JSON on every delta
Partial argument strings are not valid JSON until response.function_call_arguments.done. Accumulate for display; parse only when done (or use the final event.arguments).

Save event.response.id from response.completed as previous_response_id for turn 2 — same as the blocking loop.

Turn 2 — stream the answer

After your handler returns function_call_output, open a second stream with stream: true and handle response.output_text.delta exactly like Lesson 2. 

const turn2 = await client.responses.create({
  model: "gpt-4.1-nano",
  previous_response_id: responseId,
  input: [{
    type: "function_call_output",
    call_id: callId,
    output: toolResult,
  }],
  stream: true,
});

for await (const event of turn2) {
  if (event.type === "response.output_text.delta") {
    process.stdout.write(event.delta);
  }
}
Built-in tools
Built-in tool progress (e.g. web_search_call) also streams as output items — same event types, different item.type. Your UI can show “Searching the web…” while the platform runs the tool inside turn 1.
