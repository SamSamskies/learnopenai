Remote MCP

Lesson 7: you run the handler. Lesson 8: OpenAI runs web search. Remote MCP: someone else's server runs the tools — you just wire the URL.

Model Context Protocol (MCP) is a standard for exposing tools to AI clients. With the Responses API you can attach a remote MCP server as a built-in tool — same tools array as web_search, but OpenAI discovers tools from an external URL and calls them for you. Remote MCP guide.

Three tool patterns (recap)

Pattern type Who runs it
Custom function function Your server (Lesson 7)
Platform built-in web_search, etc. OpenAI (Lesson 8)
Remote MCP mcp External MCP server (this lesson)

Product rule: custom for your DB and auth; built-in for generic platform capabilities; MCP when a vendor or community already ships a maintained tool server (GitHub, Stripe, internal services). See remote MCP reference.

Minimal request

Point at a public MCP endpoint. OpenAI lists tools, the model may call one, and you get the answer in a single response — no function_call_output turn. 

const response = await client.responses.create({
  model: "gpt-5",
  tools: [{
    type: "mcp",
    server_label: "deepwiki",
    server_description: "MCP spec docs via DeepWiki.",
    server_url: "https://mcp.deepwiki.com/mcp",
    require_approval: {
      never: { tool_names: ["ask_question", "read_wiki_structure"] },
    },
  }],
  input: "What transport protocols does the MCP spec support?",
});

console.log(response.output_text);

The practice script uses DeepWiki (reliable, no auth). Older docs also show a dice demo at dmcp-server.deno.dev — it is often slow or down. Real servers often need authorization (OAuth bearer token).

What appears in output

type When
mcp_list_tools API imported tool definitions from the server
mcp_call Model invoked a tool — see arguments and output
message Final visible answer in output_text

Keep mcp_list_tools in context on follow-up turns (e.g. via previous_response_id) so the API does not re-fetch the tool list every call — lower latency.

Approvals and trust

Trust the MCP server
A malicious server can exfiltrate anything in the model's context. Only connect servers you trust. In production, use require_approval: "always" (or per-tool allowlists) for sensitive actions — the API returns an approval request you resolve before the call runs.

require_approval: "never" is fine for demos and low-risk tools you fully control. Default to approvals for anything that mutates data or spends money.

MCP vs custom function

Custom function Remote MCP
You implement Handler + schema per tool Nothing — server exposes tools
Turns Two (call + output) One (platform orchestrates)
Best for Your business logic, tight control Reuse vendor/community tool servers
