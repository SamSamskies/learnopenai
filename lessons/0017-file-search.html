File Search

Your docs live in a vector store. OpenAI runs the retrieval.

Lesson 16 mapped the capstone to UI phases. The capstone used a custom function against a hard-coded backlog. Real products also need answers from uploaded documents — PRDs, support articles, contracts. file_search is OpenAI's hosted RAG: you upload files, the platform indexes and retrieves them. Same one-request pattern as web search in Lesson 8. File search guide.

Setup: vector store pipeline

Unlike web_search, file search needs a one-time (or periodic) setup before you can query:

  1. Upload — files.create with purpose: "assistants"
  2. Create vector store — a named knowledge base
  3. Attach file — vectorStores.files.create
  4. Wait — poll until indexing status === "completed"
  5. Query — pass vector_store_ids in the tool config
// 1–4: setup (run once, reuse the vector store id)
const file = await client.files.create({
  file: fs.createReadStream("roadmap.pdf"),
  purpose: "assistants",
});
const store = await client.vectorStores.create({ name: "product-docs" });
await client.vectorStores.files.create(store.id, { file_id: file.id });
// … poll until completed …

// 5: query — one request, no handler
const response = await client.responses.create({
  model: "gpt-4.1-mini",
  input: "What is dark mode's priority on the roadmap?",
  tools: [{
    type: "file_search",
    vector_store_ids: [store.id],
    max_num_results: 3,
  }],
});
Pricing (docs)
Two charges on top of normal model tokens:
Storage — $0.10 / GB / day (first 1 GB free).
Tool calls — $2.50 / 1k file_search_calls (~$0.0025 each).
Tokens still bill separately — question, retrieved chunks, and answer all count.
For practice, save FILE_SEARCH_VECTOR_STORE_ID in .env and reuse it. A tiny roadmap doc stays under the free storage tier.
Other providers
For budget-conscious doc-Q&A experiments, Google Gemini File Search offers a more generous free tier (free storage, no per-search tool fee). Supabase pgvector and Pinecone/Qdrant free tiers are cheaper still if you write the retrieval loop yourself. See file search reference — Hosted RAG across providers.

Model choice

Retrieval (chunking, embedding, search) runs on OpenAI's platform. Your model decides whether to search, formulates queries, reads retrieved chunks, and writes the answer. gpt-4.1-mini is the right default for simple doc Q&A — the practice script uses it.

gpt-4.1-mini is enough when… Step up (gpt-4.1, gpt-5) when…
Short, well-structured docs Dense technical or legal content
Factual lookups (status, dates, names) Multi-hop questions across many chunks
One clear answer per question Synthesis across conflicting sources
Low stakes — user can verify citations High accuracy bar — paraphrase errors are costly

Rule of thumb: start with mini on your real docs and questions. If answers miss nuance or mash sources, bump the model before rebuilding RAG. Tune max_num_results and doc quality first — bad chunks sent to a bigger model still produce bad answers.

What's in the response

Mirror web search: a platform call item plus a message. Citations point at your files, not URLs.

Item Contains
file_search_call status, semantic queries the model used
message output_text plus file_citation annotations 
const search = response.output.find(i => i.type === "file_search_call");
const msg = response.output.find(i => i.type === "message");

search?.queries;              // ["dark mode priority roadmap"]
response.output_text;         // user-facing answer
msg?.content[0].annotations;  // [{ type: "file_citation", filename, file_id }]
UI mapping
Show “Searching documents…” while the request runs (built-ins can take a few seconds). Render filename from citations so users can verify the source — same trust pattern as url_citation for web search.

A production triage assistant might use both: custom function for the live backlog, file_search over PRDs and support macros. Mix tools in the same tools array — the model picks.

Tuning knobs

  • max_num_results — fewer chunks = lower latency and token cost; may reduce answer quality
  • filters — metadata on vector store files (e.g. only category: blog)
  • include: ["file_search_call.results"] — return raw search hits for debugging or UI “sources” panels
